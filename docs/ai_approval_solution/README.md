# AI 智能审批助手实施方案

## 1. 场景目标

本方案用于实现一个基于 AI 对话的智能审批助手。系统根据传入的 `user_id` 获取用户所属公司、部门、角色、直属主管以及可使用的审批范围，再动态加载该公司的审批模板、字段规则和审批流程。用户在聊天过程中可以用自然语言发起请假、报销、采购、补卡、外出等不同审批，AI 负责识别意图、收集字段、校验信息、生成提交预览，并在用户明确确认后调用现有审批系统接口提交申请。

核心目标不是替代审批系统，而是把审批系统的入口和表单填写过程变得更自然、更高效、更可控。

## 2. 适合解决的问题

- 用户不知道某个事项应该走哪个审批。
- 用户在审批页面里找不到入口，或者审批类型太多。
- 不同公司、不同部门的审批流程不一致，人工维护和解释成本高。
- 审批表单字段多，用户经常漏填、错填。
- 员工希望通过聊天快速发起审批，而不是进入多个页面逐项填写。
- 业务希望 AI 能辅助办理，但又不能让 AI 直接误提交。

## 3. 总体实现思路

采用 `LangGraph + 大模型 + 审批系统接口 + 向量数据库` 的组合实现。

整体流程如下：

1. 前端或业务系统传入当前登录用户的 `user_id`。
2. 后端根据 `user_id` 查询员工上下文，包括公司、部门、角色、主管、权限。
3. 系统根据员工上下文获取该用户可发起的审批模板列表。
4. 用户在聊天中提出需求，例如“我要请假”“帮我报销 2000 元”“采购两台电脑”。
5. `LangGraph` 根据当前状态进入不同节点：意图识别、审批匹配、字段收集、字段校验、预览确认、提交审批、取消或修改。
6. 大模型负责理解自然语言、抽取字段、生成追问文案和解释说明。
7. 审批模板、制度说明、字段解释等内容可以放入向量数据库，用于 RAG 检索，辅助 AI 回答“这个审批需要什么材料”等问题。
8. 字段完整后，系统生成审批预览，用户必须确认后才能正式提交。
9. 提交后返回审批编号、当前状态、审批节点和查看入口。

## 4. 技术架构

建议拆成以下几层：

### 4.1 前端入口层

正式方案不建议使用 `Streamlit` 作为用户侧页面。当前已有 Vue2 审批页面，因此第一阶段应优先复用现有页面，只在现有页面中接入 AI 助手入口、审批预览和确认操作。`Streamlit` 只适合开发人员内部快速验证节点流程和接口联调。

职责：

- 展示 AI 聊天窗口。
- 传入当前登录用户 `user_id`。
- 展示 AI 追问、审批预览、确认按钮或确认指令。
- 展示提交结果和审批编号。
- 支持用户取消、修改、重新发起。
- 复用现有 Vue2 页面风格，审批预览、字段修改、确认提交等关键动作要有明确的视觉层级。

建议页面形态：

- 现有 Vue2 审批页面接入：第一阶段优先选择，改造成本低，用户学习成本低。
- 企业工作台内嵌页：如果现有页面本身就在工作台中，则沿用当前入口。
- 飞书/企微机器人 + 卡片：适合后续移动端轻量办理，第一阶段暂不优先。

### 4.2 AI 编排层

建议使用 `LangGraph`。

职责：

- 管理审批对话状态。
- 控制节点流转。
- 防止流程混乱。
- 支持回退、修改、取消。
- 确保没有用户确认时不提交。

### 4.3 模型能力层

建议使用 DeepSeek 模型。

可选模型：

- `deepseek-chat`：适合普通对话、意图识别、字段抽取、文案生成。
- `deepseek-reasoner`：适合复杂流程推理、审批规则解释、条件分支分析，但成本和耗时可能更高。

建议第一阶段使用 `deepseek-chat`，复杂规则判断尽量交给后端确定性代码和审批模板，不完全依赖模型推理。

### 4.4 审批业务接口层

对接现有审批系统。

需要提供：

- 用户上下文接口。
- 可用审批模板接口。
- 审批模板详情接口。
- 字段预校验接口。
- 创建审批申请接口。
- 查询审批状态接口。
- 查询假期余额、报销规则等辅助接口。

### 4.5 知识库与向量数据库层

用于存储和检索制度文档、审批说明、字段解释、FAQ。

建议可选：

- `Chroma`：适合本地 Demo，轻量，上手快。
- `Milvus`：适合企业级向量检索，数据量大时更稳。
- `pgvector`：如果公司已有 PostgreSQL，可以直接扩展。
- `Qdrant`：部署简单，适合中小规模知识库。

第一阶段建议使用 `Chroma` 做 Demo；正式上线再根据公司技术栈选择 `Milvus`、`pgvector` 或 `Qdrant`。

## 5. LangGraph 节点设计

建议把审批对话拆成以下节点。

### 5.1 `load_user_context`：加载用户上下文

输入：

- `user_id`

输出：

- `company_id`
- `dept_id`
- `role`
- `manager_id`
- `available_permissions`

作用：

- 确认用户身份。
- 获取用户所属公司。
- 后续所有审批模板都根据这个上下文加载。

### 5.2 `load_approval_templates`：加载审批模板

输入：

- `company_id`
- `user_id`
- `role`

输出：

- 用户可发起的审批列表。
- 每个审批的基础信息，例如审批名称、分类、编码。

示例：

```json
[
  {
    "approval_type": "leave",
    "name": "请假申请",
    "category": "人事",
    "enabled": true
  },
  {
    "approval_type": "expense",
    "name": "报销申请",
    "category": "财务",
    "enabled": true
  }
]
```

### 5.3 `classify_intent`：识别用户意图

判断用户当前是在：

- 查询审批。
- 发起审批。
- 补充字段。
- 修改字段。
- 取消申请。
- 确认提交。
- 查询制度或说明。

注意：

意图识别不能只依赖大模型。建议规则优先，模型兜底。

例如：

- 用户说“确认提交”，必须进入确认提交节点。
- 用户说“取消”“不办了”，必须进入取消节点。
- 用户说“把金额改成 3000”，必须进入修改节点。

### 5.4 `match_approval_type`：匹配审批类型

根据用户表达和可用审批模板，判断进入哪类审批。

示例：

- “我要请年假” -> 请假申请。
- “报销客户招待费” -> 报销申请。
- “买两台电脑” -> 采购申请。
- “忘记打卡了” -> 补卡申请。

如果不确定，应返回澄清问题：

> 你是想发起“外出申请”，还是“出差申请”？

### 5.5 `collect_fields`：字段收集

根据审批模板中的字段配置逐项收集。

字段配置示例：

```json
{
  "approval_type": "expense",
  "fields": [
    {
      "name": "expense_type",
      "label": "报销类型",
      "type": "enum",
      "required": true,
      "options": ["差旅费", "餐饮费", "办公用品"]
    },
    {
      "name": "amount",
      "label": "报销金额",
      "type": "number",
      "required": true
    },
    {
      "name": "reason",
      "label": "报销事由",
      "type": "text",
      "required": true
    }
  ]
}
```

字段收集原则：

- 缺什么问什么。
- 一次只追问一个关键字段。
- 枚举字段必须校验合法选项。
- 日期、金额、数量等字段必须做格式校验。
- 不要因为用户提到其他审批关键词就随便切换流程。

### 5.6 `validate_fields`：字段校验

校验方式分两类：

#### 本地校验

- 必填字段是否为空。
- 金额是否为数字。
- 日期格式是否正确。
- 开始时间是否早于结束时间。
- 枚举值是否合法。

#### 业务接口校验

- 假期余额是否足够。
- 报销金额是否超过上限。
- 是否需要附件。
- 是否存在重复申请。
- 当前用户是否有权限提交。

### 5.7 `preview_application`：生成审批预览

在正式提交前生成预览。

预览内容建议包括：

- 审批类型。
- 申请人。
- 所属公司/部门。
- 已填写字段。
- 附件情况。
- 预计审批节点。
- 可能的风险提示。

示例：

```text
请确认是否提交报销申请：

- 报销类型：餐饮费
- 报销金额：2000 元
- 报销事由：客户招待
- 发票情况：已提供
- 预计审批节点：直属主管审批

回复“确认提交”后我再提交申请。
也可以回复“修改金额为 1800”或“取消”。
```

### 5.8 `modify_fields`：修改字段

用户可以随时修改。

支持表达：

- “金额改成 3000。”
- “结束时间改到周五。”
- “原因改成客户拜访。”
- “重新选择审批类型。”

处理方式：

1. 识别修改字段。
2. 更新状态中的字段。
3. 重新执行字段校验。
4. 重新生成审批预览。
5. 再次等待用户确认。

### 5.9 `confirm_submit`：确认提交

只有满足以下条件才允许进入提交：

- 审批类型明确。
- 必填字段完整。
- 字段校验通过。
- 用户明确确认。

用户只说“嗯”“好的”不建议直接提交，最好要求明确确认。

### 5.10 `submit_application`：提交审批

调用审批系统创建申请接口。

输入：

- `user_id`
- `approval_type`
- `form_data`
- `company_id`

输出：

- `request_id`
- `status`
- `approval_node`
- `detail_url`

### 5.11 `cancel_flow`：取消流程

用户说：

- “取消。”
- “不办了。”
- “先不提交。”
- “算了。”

系统应清空当前审批状态，并返回：

> 已取消本次审批申请，没有提交任何内容。

### 5.12 `rag_answer`：制度和流程问答

当用户问：

- “报销需要发票吗？”
- “请年假有什么规则？”
- “外出申请和出差申请有什么区别？”

系统从向量数据库检索制度文档和审批说明，再由模型生成回答。

## 6. 状态设计

`LangGraph` 中建议维护如下状态：

```python
class ApprovalState(TypedDict, total=False):
    user_id: str
    company_id: str
    dept_id: str
    role: str
    user_message: str
    intent: str
    approval_type: str | None
    available_approvals: list[dict]
    approval_template: dict | None
    slots: dict[str, str]
    awaiting_field: str | None
    errors: list[str]
    preview: str
    confirmed: bool
    request_id: str | None
    approval_node: str | None
    status: str
```

核心状态说明：

- `approval_type`：当前审批类型。
- `slots`：已收集字段。
- `awaiting_field`：当前正在等待用户补充的字段。
- `confirmed`：用户是否确认提交。
- `status`：当前流程状态，例如 `idle`、`collecting`、`awaiting_confirmation`、`submitted`、`cancelled`。

## 7. 推荐模型与组件

### 7.1 大语言模型

第一阶段建议：

- 主模型：`deepseek-chat`
- 用途：意图识别、字段抽取、追问生成、制度问答。

第二阶段可选：

- 推理模型：`deepseek-reasoner`
- 用途：复杂审批规则解释、多条件审批分支分析。

注意：

审批提交这类高风险动作不要只靠模型判断，必须由确定性规则和接口校验兜底。

### 7.2 Embedding 模型

用于审批制度、FAQ、字段说明的向量化。

可选：

- OpenAI Embeddings。
- BGE 中文向量模型。
- 通义/智谱等国产 Embedding。
- 如果公司已有统一 Embedding 服务，优先使用公司服务。

选择标准：

- 中文效果好。
- 成本可控。
- 支持批量向量化。
- 便于私有化或企业环境部署。

### 7.3 向量数据库

Demo 阶段：

- `Chroma`

正式阶段：

- `Milvus`
- `Qdrant`
- `pgvector`

选择建议：

- 如果只是几十到几百份制度文档，`Chroma` 或 `pgvector` 足够。
- 如果多公司、多制度、多版本文档较多，建议使用 `Milvus` 或 `Qdrant`。
- 如果公司数据库体系已经有 PostgreSQL，`pgvector` 运维成本最低。

### 7.4 框架组件

- `LangGraph`：审批流程编排和状态管理。
- `LangChain`：模型调用、工具封装、RAG 检索。
- `LangSmith`：调试、追踪、评估 AI 调用链路。
- 前端页面：复用现有 Vue2 审批页面。
- 办公平台卡片：如飞书卡片、企微应用消息卡片，可作为后续移动端扩展。
- `FastAPI`：正式后端服务入口。
- `Redis`：会话状态缓存，可选。
- `PostgreSQL/MySQL`：审批日志、配置快照、提交记录。

## 9. 接口清单

### 9.1 获取用户上下文

```http
GET /api/users/{user_id}/context
```

返回：

```json
{
  "user_id": "U001",
  "name": "张三",
  "company_id": "C001",
  "dept_id": "D001",
  "role": "employee",
  "manager_id": "U002"
}
```

### 9.2 获取可用审批列表

```http
GET /api/approvals/templates?user_id=U001&company_id=C001
```

返回用户可发起的审批类型。

### 9.3 获取审批模板详情

```http
GET /api/approvals/templates/{approval_type}?company_id=C001
```

返回字段配置、校验规则和审批流规则。

### 9.4 审批预校验

```http
POST /api/approvals/validate
```

用途：

- 校验字段是否合法。
- 判断余额是否足够。
- 计算预计审批节点。
- 判断是否需要附件。

### 9.5 创建审批申请

```http
POST /api/approvals
```

注意：

只有用户确认后才能调用。

### 9.6 查询审批状态

```http
GET /api/approvals/{request_id}
```

用于用户询问：

- “我刚才提交的审批到哪了？”
- “我的报销审批通过了吗？”

## 10. RAG 知识库设计

向量库主要用于回答制度和流程解释，不建议用于决定是否提交审批。

### 10.1 文档类型

- 公司制度文档。
- 审批模板说明。
- 字段填写说明。
- 常见问题。
- 异常处理规则。

### 10.2 元数据设计

每个文档块建议带上元数据：

```json
{
  "company_id": "C001",
  "approval_type": "expense",
  "doc_type": "policy",
  "version": "2026-01",
  "source": "财务报销制度",
  "effective_date": "2026-01-01"
}
```

### 10.3 检索策略

- 先按 `company_id` 过滤。
- 再按 `approval_type` 过滤。
- 再做向量相似度检索。
- 返回前 3-5 条相关内容给模型总结。

### 10.4 注意事项

- RAG 只能辅助解释，不能绕过审批系统规则。
- 如果知识库和审批模板冲突，以审批系统接口返回的规则为准。
- 制度文档需要版本管理，避免旧制度误导用户。

## 11. 用户回退与修改方案

用户可以在任意阶段修改或取消。

### 11.1 字段收集阶段

用户可以说：

- “金额改成 3000。”
- “原因换成客户拜访。”
- “我不想报销了。”

处理：

- 识别修改意图。
- 更新 `slots`。
- 重新校验。
- 继续追问缺失字段。

### 11.2 预览确认阶段

用户可以说：

- “确认提交。”
- “取消。”
- “把结束时间改到周五。”
- “重新选择审批类型。”

处理：

- 确认提交：进入提交节点。
- 取消：进入取消节点。
- 修改字段：回到字段修改节点。
- 重新选择审批类型：清空当前字段，重新匹配审批模板。

### 11.3 提交后

提交成功后不建议让 AI 直接修改原申请，除非审批系统支持撤回或变更。

可选能力：

- 查询审批状态。
- 撤回申请。
- 复制原申请重新发起。

## 12. 安全与风控

必须控制以下风险：

- AI 不允许代替用户确认提交。
- AI 不允许提交用户无权限的审批。
- AI 不允许伪造用户 ID。
- 审批提交必须由后端再次校验权限。
- 所有提交前必须有预览。
- 所有敏感操作必须记录日志。
- 模型输出不能直接作为最终业务事实，必须经过接口校验。

建议增加以下保护：

- 明确确认词：只有“确认提交”“提交申请”等明确表达才提交。
- 高风险字段二次确认：金额、日期、审批类型变更后重新预览。
- 接口超时不提交。
- 模型识别不确定时不提交，转为澄清。
- 提交失败时展示错误原因，不自动重试多次。

## 13. 评估指标

第一阶段可以评估以下指标：

- 审批发起平均耗时。
- 字段一次收集成功率。
- 审批类型识别准确率。
- 提交成功率。
- 用户修改次数。
- 用户取消率。
- AI 追问轮数。
- 误流程率。
- 人工咨询减少量。
- 用户满意度。

建议目标：

- 审批发起耗时降低 30% 以上。
- 审批类型识别准确率达到 90% 以上。
- 提交成功率达到 95% 以上。
- 误流程率低于 5%。

## 14. 分阶段落地计划

### 第一阶段：原型验证

目标：

- 做出可演示、视觉上可接受的审批助手原型。
- 支持 2-3 类审批。
- 不直接接生产系统。

内容：

- 在现有 Vue2 审批页面中接入 AI 助手入口、聊天交互和审批预览。
- 使用 `LangGraph` 编排流程。
- 使用 DeepSeek 做意图识别和字段抽取。
- 使用 mock 接口模拟用户、审批模板和提交结果。
- 使用 Chroma 做制度问答原型。

说明：

`Streamlit` 可以作为开发人员内部调试工具，但不作为对业务方展示或最终交付页面。对外展示版本优先复用现有 Vue2 页面，并至少具备 AI 入口、对话区、审批预览、字段修改、确认/取消、提交结果反馈等能力。

建议审批类型：

- 请假申请。
- 报销申请。
- 采购申请。

### 第二阶段：测试环境联调

目标：

- 接入真实审批测试环境。
- 验证真实接口、字段、权限。

内容：

- 对接用户上下文接口。
- 对接审批模板接口。
- 对接预校验接口。
- 对接创建审批接口。
- 增加日志和错误处理。
- 使用 LangSmith 跟踪模型调用和节点流转。

### 第三阶段：小范围试点

目标：

- 找一个公司或部门试用。
- 收集真实使用数据。

内容：

- 限定用户范围。
- 限定审批类型。
- 增加人工兜底入口。
- 统计效果指标。
- 根据反馈优化追问文案、字段抽取和流程切换。

### 第四阶段：扩展更多审批

目标：

- 从 2-3 类审批扩展到更多审批。

内容：

- 建立审批模板标准。
- 建立制度知识库维护流程。
- 支持多公司模板。
- 支持更多复杂条件分支。
- 增加审批状态查询、撤回、复制再发起等能力。

## 15. 推荐第一版技术选型

第一版建议选型：

- 前端页面：复用现有 Vue2 审批页面
- 移动端/办公入口：飞书卡片、企微卡片可作为后续扩展，第一阶段暂不优先
- 后端服务：`FastAPI`
- 流程编排：`LangGraph`
- LLM：`DeepSeek Chat`
- RAG 框架：`LangChain`
- 向量数据库：`Chroma`
- 观察与调试：`LangSmith`
- 会话状态：内存或 Redis
- 业务数据：mock 接口，后续替换为真实审批系统接口

不建议：

- 不建议用 `Streamlit` 作为正式页面。
- 不建议当前阶段重做前端页面或升级前端技术栈。
- 不建议只做命令行或纯后端 Demo，因为该场景需要验证用户确认、字段修改、审批预览等交互体验。

如果要尽快做出可演示版本，可以先不接向量数据库，只做审批发起闭环。等审批流程稳定后，再补制度问答和 RAG。

## 16. 最小可用版本范围

建议 MVP 只做以下能力：

1. 传入 `user_id`。
2. mock 返回用户所属公司。
3. mock 返回该公司可用审批模板。
4. 支持请假、报销、采购三类审批。
5. 支持自然语言发起。
6. 支持缺字段追问。
7. 支持字段修改。
8. 支持取消。
9. 支持提交前预览。
10. 支持用户确认后 mock 提交。
11. 在现有 Vue2 页面中接入 AI 助手入口，支持审批预览、字段修改和确认提交。

MVP 暂不做：

- 复杂附件上传。
- 真实生产审批提交。
- 多轮审批状态追踪。
- 大规模制度知识库。
- 多语言。
- 复杂权限模型。

## 17. 结论

该场景适合使用 `LangGraph`。因为审批办理不是单轮问答，而是一个有状态、有节点、有回退、有确认、有提交动作的流程。`LangGraph` 可以把大模型的自然语言能力和业务系统的确定性规则结合起来：模型负责理解和表达，图负责流程控制，审批系统负责权限、校验和提交。

第一阶段建议以“请假 + 报销 + 采购”作为试点，先跑通 `user_id -> 公司审批模板 -> 对话收集字段 -> 预览确认 -> 提交系统` 的闭环。等试点证明效率提升、误流程率可控后，再扩展更多审批类型和制度问答能力。
